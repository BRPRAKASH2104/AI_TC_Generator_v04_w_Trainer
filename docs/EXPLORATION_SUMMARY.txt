PROCESSING WORKFLOWS
================================================================================

STANDARD PROCESSING:
File: src/processors/standard_processor.py
Class: REQIFZFileProcessor (extends BaseProcessor)
Method: process_file(reqifz_path, model, template, output_dir)

Flow:
1. Initialize logger and components
2. Extract artifacts (sequential)
3. Build augmented requirements with context
4. For each requirement:
   - Build prompt
   - Call LLM (blocks until response)
   - Parse response
   - Add to results
5. Format to Excel
6. Save RAFT examples (if enabled)
7. Return result

Performance:
- Sequential processing: one requirement at a time
- Throughput: ~7 test cases/second
- Good for: debugging, small files (< 100 requirements)
- Memory: All artifacts loaded in memory

HIGH-PERFORMANCE (ASYNC) PROCESSING:
File: src/processors/hp_processor.py
Class: HighPerformanceREQIFZFileProcessor (extends BaseProcessor)
Method: async process_file(reqifz_path, model, template, output_dir)

Flow:
1. Extract artifacts (parallel with ThreadPoolExecutor)
2. Build augmented requirements with context
3. Create async tasks for ALL requirements
4. Execute concurrently with:
   - AsyncOllamaClient (rate-limited by semaphore)
   - asyncio.gather() with exception handling
5. Stream to Excel (memory-efficient)
6. Save RAFT examples (if enabled)
7. Return result with performance metrics

Performance:
- Concurrent processing: all requirements in flight simultaneously
- Throughput: ~24 test cases/second (3-5x improvement)
- Good for: large files (> 100 requirements)
- Memory: Streaming to avoid accumulation
- Concurrency: 2 GPU requests (semaphore) + queue

================================================================================
CRITICAL ARCHITECTURAL INSIGHTS
================================================================================

1. CONTEXT-AWARE PROCESSING IS THE HEART
   The system's primary innovation is BaseProcessor._build_augmented_requirements()
   which prepares rich context for each requirement:
   - Section heading (where this requirement belongs)
   - Information artifacts (notes/guidelines for this section)
   - System interfaces (global signal/parameter dictionary)
   
   This context dramatically improves LLM-generated test case quality.

2. DUAL PROCESSING MODES FOR FLEXIBILITY
   - Standard: Sequential, good for understanding/debugging
   - HP: Concurrent/async, 3-5x faster for large datasets
   
   Both share identical core logic via BaseProcessor inheritance.

3. MODULAR COMPONENT DESIGN
   Clean separation of concerns:
   - Extractors: REQIF parsing
   - Generators: Test case generation
   - Formatters: Output formatting
   - Parsers: Response parsing
   - OllamaClient: LLM API interaction
   - PromptBuilder: Prompt construction
   
   Each component is independently testable and replaceable.

4. STRUCTURED ERROR HANDLING
   Custom exception hierarchy with actionable context:
   - Connection errors include host/port
   - Timeout errors include timeout value
   - Model errors include model name
   - Response errors include status code and body
   
   Enables 10x faster debugging.

5. ATTRIBUTE DEFINITION MAPPING (v2.0 FIX)
   Critical fix that enables extraction:
   - REQIF XML uses internal identifiers for attributes
   - Must map "_json2reqif_XXX" to "ReqIF.Text" LONG-NAME
   - Without this mapping: 0% extraction success
   - With this mapping: 100% extraction success
   
   This mapping is built in _build_attribute_definition_mapping() (lines 157-180)

6. PERFORMANCE OPTIMIZATION LAYERS
   - __slots__ for 20-30% memory reduction (24/32 classes)
   - Streaming XML parsing for large files (memory-efficient)
   - Concurrent extraction with ThreadPoolExecutor
   - Concurrent LLM calls with asyncio + semaphore
   - Single semaphore in AsyncOllamaClient (not in generator - v1.5.0 fix)

7. TEMPLATE-BASED PROMPTING
   YAML templates allow:
   - Easy prompt customization without code changes
   - Multiple template support for different requirement types
   - Adaptive template that handles both table and text-only requirements
   - Variable substitution for context injection
